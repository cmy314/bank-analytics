# 多分片固定微服务/实例 — 拉长时间序列

目标：在 **同一 `msname`、同一 `msinstanceid`** 上，把 `MSRTQps_0`+`MSResource_0`、`_1`、`_2`… 拼成 **一条更长的时间序列**，再训练 IF 与短窗容量。

---

## 一、准备数据

1. 解压 Alibaba v2021，得到同一目录下的配对文件（**分片号必须一致**）：

   ```
   your-data-dir/
     MSRTQps_0.csv      MSResource_0.csv
     MSRTQps_1.csv      MSResource_1.csv
     MSRTQps_2.csv      MSResource_2.csv   （可选）
   ```

2. **不要**混用 `MSRTQps_1` 与 `MSResource_0`。

3. 内存建议：每片单独 merge 时仍可用 `MSRT_NROWS` / `MS_RESOURCE_NROWS`；拼接后在 **单实例** 上训练，IF 阶段内存较小。整表 pivot 建议 **8GB+ RAM** 或 Kaggle（见 [云端运行v2021指南.md](云端运行v2021指南.md)）。

---

## 二、配置 `.env`

在 `analytics/.env` 中增加（路径按你的机器修改）：

```env
# CSV 所在目录（分片文件均在此目录）
DATA_DIR_V2021=E:/data/alibaba-v2021

# 分片 0 仍可作为默认单文件路径（可选，与 DATA_DIR 一致即可）
MSRT_PATH=E:/data/alibaba-v2021/MSRTQps_0.csv
MS_RESOURCE_PATH=E:/data/alibaba-v2021/MSResource_0.csv

OUTPUT_DIR_V2021=E:/output/v2021_multishard

# 分片 0 用「调用量最大的 msname」；分片 1+ 自动沿用 anchor
ONLY_FIRST_MSNAME=true

# 可选：分片 0 起就固定微服务（跨分片必须一致时再设）
# MSNAME_FILTER=完整msname字符串

# 可选：手动指定实例（否则取 merged 中行数最多的 msinstanceid）
# MSINSTANCEID_FILTER=完整msinstanceid字符串

# 命令行默认分片列表（也可 --shards 覆盖）
V2021_SHARDS=0,1,2

# 调试时可限制行数；正式实验留空读全片
# MSRT_NROWS=800000
# MS_RESOURCE_NROWS=2000000
```

## 三、一条命令跑完全流程

```powershell
cd F:\bank-observability-demo\analytics
.\.venv\Scripts\Activate.ps1
python -m bank_analytics v2021-shards --shards 0,1,2
```

或右键 `run_v2021_shards.py`（读取环境变量 `V2021_SHARDS`）。

### 流程说明

| 步骤 | 行为 |
|------|------|
| 分片 0 | merge → 写入 `output/.../shards/shard_0/merged_v2021.*` → 确定 **anchor** |
| 分片 1、2… | 仅用 anchor 的 **msname** 过滤后 merge |
| 拼接 | 按 `(timestamp, msname, msinstanceid)` 去重 → `merged_v2021_multishard.parquet` |
| 建模 | 固定 anchor 的 **msinstanceid** → IF + P95 + 短窗容量 |

anchor 会保存为：

`OUTPUT_DIR_V2021/shard_anchor.json`

示例：

```json
{
  "msname": "...",
  "msinstanceid": "...",
  "shard_indices": [0, 1, 2]
}
```

**第二遍加分片**时，可把 `shard_anchor.json` 里的值抄到 `.env`：

```env
MSNAME_FILTER=...
MSINSTANCEID_FILTER=...
```

避免分片 1 里「调用量最大」变成别的服务。

## 四、输出文件

| 路径 | 含义 |
|------|------|
| `shards/shard_N/merged_v2021.*` | 各分片宽表（多实例） |
| `merged_v2021_multishard.parquet` | 拼接后的宽表 |
| `shard_anchor.json` | 固定的 msname / msinstanceid |
| `if_v2021.joblib`、`fig_v2021_*.png` | 在 **拼接后的单实例序列** 上训练 |
| `capacity_*` | 同上 |

日志中关注：

```
拼接后单实例时间序列: XXXX 行, ds [t0 .. t1]
```

`XXXX` 应明显大于只跑 `_0` 时的行数。

## 六、常见问题

**Q：分片 1 merge 后行数为 0？**  
A：`msname` 在该时间窗不存在或写错。用 `shard_anchor.json` 里的完整字符串设 `MSNAME_FILTER`。

**Q：拼接后行数没增加多少？**  
A：检查两片的 `ds` 时间范围是否衔接；或 `MSRT_NROWS` 截断过早。

**Q：和单分片 `python -m bank_analytics v2021` 的区别？**  
A：单分片只训 `_0` 时间窗；`v2021-shards` 用 **同一实例** 跨分片延长序列，模型与图写在 `OUTPUT_DIR_V2021` 根目录。

**Q：论文怎么写？**  
A：实验设计可写为「选取负载最高的微服务及其代表实例，在公开 trace 的连续时间分片上合并观测，构成更长四维时间序列」。

---

## 七、推荐实验组合

1. **基线**：`v2021` 仅 `_0`。  
2. **加长**：`v2021-shards --shards 0,1` 或 `0,1,2`。  
3. **对比**：对比两次的 `instance_df` 行数、IF 告警率、容量系数 α/β。
